# The GGEN-Only User Surface (The Covenant)

> Explanation — the single governing rule of `crates/wasm4games`. Read this before you
> touch the crate. For background, see the [foundry overview](wasm4games-overview.md); to
> see the rule in action, follow the
> [add-a-pattern tutorial](../tutorials/wasm4games-add-a-pattern.md).

## The rule, plainly

**In `wasm4games`, you edit the ontology and run a generator. You never hand-write a
kernel.**

Concretely, there are exactly two things you are permitted to author by hand:

1. The RDF ontology and its generation inputs under
   **`crates/wasm4games/ggen/`** — `schema/patterns.ttl` (the pattern declarations),
   `queries/*.rq` (the SPARQL that extracts them), and `templates/*.tera` (the Rust the
   templates emit).
2. The hand-written *substrate* it builds on: `class.rs`, `ir.rs`, the `evidence/`
   types, `verify.rs`, and `compat.rs` — the stable data model that the generated code
   targets.

Everything under **`crates/wasm4games/src/patterns/`** — every kernel and the
`PATTERN_REGISTRY` — is **generated output**. You regenerate it; you do not edit it.

```
                      YOU EDIT THIS                         YOU NEVER EDIT THIS
        ┌────────────────────────────────────┐     ┌──────────────────────────────┐
        │ crates/wasm4games/ggen/             │     │ crates/wasm4games/src/        │
        │   schema/patterns.ttl   (the law)   │     │   patterns/                   │
        │   queries/*.rq          (SPARQL)    │ ──► │     mod.rs   (PATTERN_REGISTRY)│
        │   templates/*.tera      (Rust out)  │     │     *.rs     (the kernels)     │
        │   ggen.toml             (pipeline)  │     │   ▲ GENERATED BY ggen          │
        └────────────────────────────────────┘     └──────────────────────────────┘
                                  └────────── ggen sync ──────────┘
```

`ggen` is the deterministic code generator at <https://github.com/seanchatmangpt/ggen>.

## Why a covenant and not a convention

This is stated as a *covenant* because the entire value proposition of `wasm4games`
collapses if kernels drift by hand:

- **The ontology is the source of truth.** `patterns.ttl` is what `ir::PatternSpec`,
  `PATTERN_REGISTRY`, and `verify` all mirror. A hand-edited kernel makes the registry lie
  about what the system does, and the self-checks then verify a fiction.
- **Determinism is the product.** Patterns claim deterministic, auditable behavior. If the
  generated layer can be quietly mutated, "deterministic" becomes unprovable.
- **Verification anchors to data.** The verify oracle and the (external) wasm4pm authority
  both reason over the *declared* spec. Diverging code defeats both.

So the rule is not stylistic. Editing `src/patterns/*` by hand is a defect on par with
editing a compiler's generated output and then claiming the source compiles to it.

## The generated banner

Every generated file opens with an unmissable banner:

```rust
// GENERATED BY ggen — DO NOT EDIT
// Source: crates/wasm4games/ggen/schema/patterns.ttl
// Regenerate with: ggen sync   (then: cargo fmt -p wasm4games)
```

If you see that banner, stop. The change you want belongs in the TTL (or a query, or a
template). Make it there and regenerate; the banner is the contract that lets the next
contributor trust that `src/patterns/*` is exactly what the ontology says.

## Why generated files are committed

A natural question: if the code is generated, why is it checked into git rather than built
on demand?

Because **bcinr's CI is offline and strict.** The workspace builds under network-free,
`-D warnings` conditions (`RUSTFLAGS="-D warnings" cargo make all`). If kernels were
generated at build time, CI would need to fetch and run ggen — pulling a git tool into a
network-free pipeline and adding a build-time dependency the security posture forbids.

By **committing the generated `.rs` files**, the offline build needs *nothing but rustc*.
ggen becomes a developer-time tool, not a build-time dependency. The committed output is
the offline source of truth; ggen exists only to *regenerate* it when the ontology
changes. This mirrors the wider bcinr philosophy: the logic substrate has zero runtime
dependencies, and the generated layer adds zero build-time ones.

The trade is that **regeneration is a human step with a reviewable diff**. When you change
`patterns.ttl` and run `ggen sync`, the resulting change to `src/patterns/*` shows up in
your commit, where a reviewer can confirm the generated code matches the declared intent.

## The one post-sync step

`ggen` emits correct Rust, but template output is not guaranteed to be `rustfmt`-clean.
Because the workspace enforces formatting in CI, there is exactly **one** required step
after every sync:

```bash
ggen sync
cargo fmt -p wasm4games
```

That is the whole post-generation ritual. After formatting, the normal checks
(`cargo test -p wasm4games --features std`, `cargo clippy`, `cargo fmt --check`) apply as
to any other crate. There is no hidden hand-finishing of the generated code.

## Determinism via `ORDER BY ?id`

Code generation must be **reproducible**: running `ggen sync` twice on the same ontology
must produce byte-identical output, or the committed diff becomes noisy and review becomes
worthless.

The mechanism is simple and lives in the SPARQL layer. Every `SELECT` query that drives a
template ends with a total order on a stable key:

```sparql
SELECT ?id ?name ?lowering ?eventCode ?primitive
WHERE {
    ?p a w4g:Pattern ;
       w4g:id ?id ;
       w4g:name ?name ;
       w4g:lowering ?lowering ;
       w4g:eventCode ?eventCode ;
       w4g:primitive ?primitive .
}
ORDER BY ?id          # ← total, stable order ⇒ deterministic emission
```

RDF triples are an *unordered* set; without `ORDER BY` the generator could legitimately
emit patterns in any order, and two runs could differ. Ordering by the stable `w4g:id`
pins the emission order, so the kernels appear in the file — and in `PATTERN_REGISTRY` —
in a fixed, id-sorted sequence on every machine. Deterministic input order is what makes
the committed output a *function* of the ontology.

## The ggen pipeline

The full path from declaration to committed Rust:

```
  patterns.ttl            queries/*.rq            templates/*.tera         src/patterns/*.rs
  (RDF ontology)   ──►     (SPARQL SELECT,   ──►   (Tera render over   ──►  (committed Rust,
   the law                  ORDER BY ?id)           query rows)               banner-marked)
                                              run by: ggen sync (reads ggen.toml)
```

1. **TTL ontology** — you declare each pattern as a `w4g:Pattern` individual in
   `patterns.ttl` (id, name, lowering, event code, primitive, touched objects, admission).
2. **SPARQL SELECT** — a `.rq` query projects the ontology into ordered rows, one per
   pattern, terminating in `ORDER BY ?id` for determinism.
3. **Tera template** — a `.tera` template renders each row into a branchless kernel plus
   its `PatternSpec` entry, stamping the `// GENERATED BY ggen — DO NOT EDIT` banner.
4. **Committed `.rs`** — the output is written under `src/patterns/`, committed to git, and
   becomes the offline source of truth.

The pipeline is configured in **`ggen.toml`**, whose `[[generation.rules]]` entries bind
each query to its template and output path:

```toml
# crates/wasm4games/ggen.toml
[[generation.rules]]
name     = "pattern-kernels"
query    = "queries/patterns.rq"
template = "templates/kernel.rs.tera"
output   = "src/patterns/{{ name }}.rs"

[[generation.rules]]
name     = "pattern-registry"
query    = "queries/patterns.rq"
template = "templates/registry.rs.tera"
output   = "src/patterns/mod.rs"
```

Running `ggen sync` reads `ggen.toml`, executes each rule, and (re)writes the committed
files in `src/patterns/`.

## The contrast with normal frameworks

In a conventional game framework, the documentation teaches you to **hand-write** the
behavior: "implement this trait," "override this method," "wire this system." The code is
the artifact and the docs explain how to type it.

`wasm4games` inverts this. The docs here teach **declaration, not hand-coding**:

| Conventional framework | wasm4games |
|------------------------|-----------|
| Write a function/impl | Declare a `w4g:Pattern` in `patterns.ttl` |
| Code is the source of truth | The *ontology* is the source of truth |
| Refactor by editing code | Refactor by editing the ontology + templates |
| Tests check hand-written code | `verify` checks generated code against the spec |
| Docs explain how to type it | Docs explain how to *declare* it |

So when the rest of this documentation set shows you how to "add a pattern," it will never
show you hand-writing a kernel. It shows you editing an RDF file and running `ggen sync`.
That is not an omission — it is the covenant. The kernels are generated by ggen; you write
the law, and ggen manufactures it into branchless Rust.
